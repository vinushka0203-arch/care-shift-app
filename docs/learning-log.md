## 2026-07-03 設計立案・技術スタック決定

### 何をしたか
- MVP の範囲を「手動シフト管理」に確定し、`docs/design.md` に設計書を作成した
- 技術スタックを FastAPI + React (TypeScript) に決定した
- Git リポジトリを初期化し、GitHub(パブリック)で管理を開始した

### なぜその実装にしたか
- **FastAPI + React**: Python を学びたいため。Next.js + Supabase(1言語で完結)とも比較したが、バックエンドを自分で書く経験を優先した
- **MVP は手動シフト管理**: 自動生成(制約充足問題)は難易度が高く未完成リスクがあるため、まず確実に完成する範囲に絞った
- **開発 SQLite → 本番 PostgreSQL**: 初期のセットアップコストをゼロにして開発速度を優先。SQLAlchemy を挟むので移行コストは小さい
- **シフト保存は bulk PUT**: 管理者は表をまとめて編集するため、セル単位の POST より通信回数と整合性の面で有利
- **UNIQUE(user_id, work_date)**: 「1人1日1区分」というビジネスルールをアプリ側だけでなく DB 制約でも守る

### 学んだこと
- 設計書は「含めないもの(スコープ外)」を明記すると、途中で機能が膨らむのを防げる
- ビジネスルールは DB 制約(UNIQUE 等)で守るとバグに強くなる

## 2026-07-03 プロジェクト開始・CLAUDE.md の整備

### 何をしたか
- CLAUDE.md を作成し、学習ログの運用ルールとモデルの使い分け方針を定義した
- 実装担当のサブエージェント `implementer`(Sonnet)を `.claude/agents/` に定義した

### なぜその実装にしたか
- 学習ログを CLAUDE.md 本体に直接書き続けると肥大化し、毎セッションのコンテキストを圧迫するため、別ファイル(このファイル)に分離し、CLAUDE.md には「書き方のルール」だけを置いた
- モデルの使い分けは CLAUDE.md に書くだけでは切り替わらないため、`model:` を指定したサブエージェント定義で実現した

### 学んだこと
- CLAUDE.md はセッション開始時に毎回読み込まれる「常駐の指示書」であり、内容は短く・ルール中心に保つのが良い
- サブエージェント(`.claude/agents/*.md`)はフロントマターの `model:` でモデルを固定でき、役割ごとのモデル使い分けができる
