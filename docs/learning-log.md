## 2026-07-07 M1: 環境構築+認証(ログイン)の実装

### 何をしたか
- FastAPI バックエンドの雛形(SQLAlchemy 2.0 + Alembic + SQLite)と users テーブルを作成
- JWT 認証を実装: `POST /api/auth/login`(トークン発行)と `GET /api/users/me`(認証保護 API)
- React (Vite + TypeScript + Tailwind v4) のフロントエンド雛形と、ログイン画面・認証ガード付きトップ画面を作成
- pytest 6件(ログイン成功/失敗/無効ユーザー、/me の認証あり/なし)を整備し全件パス
- 実装は `implementer` サブエージェント(Sonnet)に委譲し、本体(Fable 5)がコードレビューを実施

### なぜその実装にしたか
- **JWT の sub にメールアドレスを採用**: users.email は unique 制約があるため識別子として機能する。MVP では十分と判断
- **テスト DB を本体と分離**(`test.db` + FastAPI の `dependency_overrides`): テストが開発データを壊さないようにするための定石
- **bcrypt を 4.0.1 にピン留め**: passlib 1.7.4 が bcrypt 4.1 以降と互換性問題を起こすため
- **Vite 雛形の不要ファイルは削除せず `_trash/` へ退避**: プロジェクトの削除禁止ルールに準拠

### 学んだこと
- **コードレビューの効果を実感**: サブエージェントの実装は設計書に忠実だったが、「`is_active=False`(退職者)でもログインできてしまう」という仕様の穴をレビューで発見。ログイン時とトークン検証時(`get_current_user`)の両方で `is_active` を確認するよう修正した。トークンは発行後 24 時間有効なため、ログイン時だけの確認では「退職処理後もトークンが生き残る」穴が残る
- FastAPI の `dependency_overrides` を使うと、テスト時だけ DB 接続を差し替えられる(DI の実用例)
- コミット前に `git check-ignore` で `.venv` / `.env` / `node_modules` / `*.db` の除外を点検すると、秘密情報や巨大フォルダの誤コミットを防げる

## 2026-07-03 設計立案・技術スタック決定

### 何をしたか
- MVP の範囲を「手動シフト管理」に確定し、`docs/design.md` に設計書を作成した
- 技術スタックを FastAPI + React (TypeScript) に決定した
- Git リポジトリを初期化し、GitHub(パブリック)で管理を開始した

### なぜその実装にしたか
- **FastAPI + React**: Python を学びたいため。Next.js + Supabase(1言語で完結)とも比較したが、バックエンドを自分で書く経験を優先した
- **MVP は手動シフト管理**: 自動生成(制約充足問題)は難易度が高く未完成リスクがあるため、まず確実に完成する範囲に絞った
- **開発 SQLite → 本番 PostgreSQL**: 初期のセットアップコストをゼロにして開発速度を優先。SQLAlchemy を挟むので移行コストは小さい
- **シフト保存は bulk PUT**: 管理者は表をまとめて編集するため、セル単位の POST より通信回数と整合性の面で有利
- **UNIQUE(user_id, work_date)**: 「1人1日1区分」というビジネスルールをアプリ側だけでなく DB 制約でも守る

### 学んだこと
- 設計書は「含めないもの(スコープ外)」を明記すると、途中で機能が膨らむのを防げる
- ビジネスルールは DB 制約(UNIQUE 等)で守るとバグに強くなる

## 2026-07-03 プロジェクト開始・CLAUDE.md の整備

### 何をしたか
- CLAUDE.md を作成し、学習ログの運用ルールとモデルの使い分け方針を定義した
- 実装担当のサブエージェント `implementer`(Sonnet)を `.claude/agents/` に定義した

### なぜその実装にしたか
- 学習ログを CLAUDE.md 本体に直接書き続けると肥大化し、毎セッションのコンテキストを圧迫するため、別ファイル(このファイル)に分離し、CLAUDE.md には「書き方のルール」だけを置いた
- モデルの使い分けは CLAUDE.md に書くだけでは切り替わらないため、`model:` を指定したサブエージェント定義で実現した

### 学んだこと
- CLAUDE.md はセッション開始時に毎回読み込まれる「常駐の指示書」であり、内容は短く・ルール中心に保つのが良い
- サブエージェント(`.claude/agents/*.md`)はフロントマターの `model:` でモデルを固定でき、役割ごとのモデル使い分けができる
