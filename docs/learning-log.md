## 2026-07-08 M3: 月間シフト表(作成・編集・閲覧)の実装

### 何をしたか
- shifts テーブル(職員×日付×勤務区分、UNIQUE(user_id, work_date))とマイグレーションを追加
- `GET /api/shifts?year=&month=`(月次取得)と `PUT /api/shifts/bulk`(一括保存)を実装
- シフト表ページを新設してトップページ化(旧挨拶画面は `_trash/` へ退避)。管理者はパレット方式で編集(区分を選んでセルをクリック)、職員は閲覧のみ
- テスト 19件 → 26件(月境界フィルタ、上書き、null 削除、権限、422)

### なぜその実装にしたか
- **保存は bulk PUT 方式**: セルを1つ塗るたびに API を叩くのではなく、編集をフロント側で溜めて「保存」でまとめて送る。通信回数が減り、「保存を押すまで確定しない」という業務ソフトらしい安心感のある UX になる。null は「セルを空に戻す」の意味に割り当てた
- **月次レスポンスに職員一覧を同梱**: `GET /api/users` は管理者専用のため、シフト表の描画に必要な最小限(id と名前)だけを月次 API に含め、一般職員も表を閲覧できるようにした。API の権限設計を崩さずに閲覧要件を満たす折衷案
- **編集の差分管理**: フロントは「サーバーの状態(baseline)」と「未保存の編集(edits)」を分けて持ち、同じ値に塗り直したら編集扱いから外す。未保存件数の表示や「破棄」ボタンが自然に実現できる
- **同一セルの重複指定は最後を採用**: bulk の中に同じセルが2回来ると UNIQUE 制約違反になるため、サーバー側で dict により後勝ちで重複排除してから処理する

### 学んだこと
- **SQLAlchemy に直接データを入れるときは型変換が働かない**: API 経由なら Pydantic が "09:00" → time 型に変換するが、テストで DB に直接 insert すると `SQLite Time type only accepts Python time objects` エラーになる。レイヤーごとに「誰が型を保証しているか」を意識する
- **バックグラウンドのサーバープロセスは残留することがある**: 前日に起動した uvicorn が生きたままポート 8000 を掴んでおり、新サーバーが Errno 10048(bind 失敗)で落ちて「古いコードに対して動作確認していた」状態になった。ポートの使用状況(`Get-NetTCPConnection`)で原因を特定。「サーバーが応答する ≠ 最新のコードが動いている」
- 月の日数は `new Date(year, month, 0).getDate()` で取れる(JS の Date は「翌月の0日目=当月末日」になる仕様)

## 2026-07-07 M2: 職員管理・勤務区分管理の実装(+開発体制の見直し)

### 何をしたか
- 管理者権限チェック(`get_current_admin`)を追加し、職員管理 API(一覧・登録・部分更新)を実装
- 勤務区分(shift_types)のモデル・マイグレーション・API を実装し、標準6区分(早番/日勤/遅番/夜勤/明け/休み)をシード投入
- フロントエンドに共通レイアウト(ヘッダー+ナビ、管理者にだけ管理メニュー表示)、職員管理ページ、勤務区分管理ページを追加
- テストを 6件 → 19件 に拡充(権限 403、email 重複 409、404、時刻バリデーション 422 など)
- 開発体制を変更: サブエージェント自動委譲をやめ、基本はメインセッション(Fable 5)が直接実装する方針に(CLAUDE.md 更新)

### なぜその実装にしたか
- **職員の削除機能は作らない**: 退職者もシフト履歴と紐づくため、削除ではなく `is_active=False` での無効化に統一。誤削除で履歴が壊れる事故を防ぐ業務システムの定石
- **認可は依存性の連鎖で表現**: `get_current_user`(認証)→ `get_current_admin`(認可)と FastAPI の Depends を重ねることで、各エンドポイントは引数に書くだけで権限制御できる
- **email は変更不可**: JWT の識別子(sub)に email を使っているため、変更を許すと発行済みトークンが宙に浮く。MVP では変更不可とした
- **エラーコードの使い分け**: 権限なし=403、email 重複=409(Conflict)、入力不正=422 と HTTP セマンティクスに沿わせ、フロントは detail メッセージをそのまま表示
- **「勤務区分に時刻必須」は Pydantic の model_validator で表現**: DB 制約では書きにくい条件付きルール(is_work=True のときだけ必須)はスキーマ層で守る

### 学んだこと
- FastAPI のルート定義は順序が重要: `GET /api/users/me` を `GET /api/users/{user_id}` より先に定義しないと、"me" が user_id として解釈されてしまう
- react-router の `<Outlet />` を使うと、ヘッダー付き共通レイアウトを1箇所で定義してネストした画面に適用できる
- TanStack Query は `invalidateQueries` で「保存後に一覧を自動再取得」が実現でき、手動での状態同期が不要になる
- サブエージェントは途中停止すると未承認の書き込みが失われることがある(「実装済み」の報告と実ファイルの不一致を経験)。作業の見える化と確実性の面から、学習用途ではメインセッションでの逐次実装が合っていた

## 2026-07-07 M1: 環境構築+認証(ログイン)の実装

### 何をしたか
- FastAPI バックエンドの雛形(SQLAlchemy 2.0 + Alembic + SQLite)と users テーブルを作成
- JWT 認証を実装: `POST /api/auth/login`(トークン発行)と `GET /api/users/me`(認証保護 API)
- React (Vite + TypeScript + Tailwind v4) のフロントエンド雛形と、ログイン画面・認証ガード付きトップ画面を作成
- pytest 6件(ログイン成功/失敗/無効ユーザー、/me の認証あり/なし)を整備し全件パス
- 実装は `implementer` サブエージェント(Sonnet)に委譲し、本体(Fable 5)がコードレビューを実施

### なぜその実装にしたか
- **JWT の sub にメールアドレスを採用**: users.email は unique 制約があるため識別子として機能する。MVP では十分と判断
- **テスト DB を本体と分離**(`test.db` + FastAPI の `dependency_overrides`): テストが開発データを壊さないようにするための定石
- **bcrypt を 4.0.1 にピン留め**: passlib 1.7.4 が bcrypt 4.1 以降と互換性問題を起こすため
- **Vite 雛形の不要ファイルは削除せず `_trash/` へ退避**: プロジェクトの削除禁止ルールに準拠

### 学んだこと
- **コードレビューの効果を実感**: サブエージェントの実装は設計書に忠実だったが、「`is_active=False`(退職者)でもログインできてしまう」という仕様の穴をレビューで発見。ログイン時とトークン検証時(`get_current_user`)の両方で `is_active` を確認するよう修正した。トークンは発行後 24 時間有効なため、ログイン時だけの確認では「退職処理後もトークンが生き残る」穴が残る
- FastAPI の `dependency_overrides` を使うと、テスト時だけ DB 接続を差し替えられる(DI の実用例)
- コミット前に `git check-ignore` で `.venv` / `.env` / `node_modules` / `*.db` の除外を点検すると、秘密情報や巨大フォルダの誤コミットを防げる

## 2026-07-03 設計立案・技術スタック決定

### 何をしたか
- MVP の範囲を「手動シフト管理」に確定し、`docs/design.md` に設計書を作成した
- 技術スタックを FastAPI + React (TypeScript) に決定した
- Git リポジトリを初期化し、GitHub(パブリック)で管理を開始した

### なぜその実装にしたか
- **FastAPI + React**: Python を学びたいため。Next.js + Supabase(1言語で完結)とも比較したが、バックエンドを自分で書く経験を優先した
- **MVP は手動シフト管理**: 自動生成(制約充足問題)は難易度が高く未完成リスクがあるため、まず確実に完成する範囲に絞った
- **開発 SQLite → 本番 PostgreSQL**: 初期のセットアップコストをゼロにして開発速度を優先。SQLAlchemy を挟むので移行コストは小さい
- **シフト保存は bulk PUT**: 管理者は表をまとめて編集するため、セル単位の POST より通信回数と整合性の面で有利
- **UNIQUE(user_id, work_date)**: 「1人1日1区分」というビジネスルールをアプリ側だけでなく DB 制約でも守る

### 学んだこと
- 設計書は「含めないもの(スコープ外)」を明記すると、途中で機能が膨らむのを防げる
- ビジネスルールは DB 制約(UNIQUE 等)で守るとバグに強くなる

## 2026-07-03 プロジェクト開始・CLAUDE.md の整備

### 何をしたか
- CLAUDE.md を作成し、学習ログの運用ルールとモデルの使い分け方針を定義した
- 実装担当のサブエージェント `implementer`(Sonnet)を `.claude/agents/` に定義した

### なぜその実装にしたか
- 学習ログを CLAUDE.md 本体に直接書き続けると肥大化し、毎セッションのコンテキストを圧迫するため、別ファイル(このファイル)に分離し、CLAUDE.md には「書き方のルール」だけを置いた
- モデルの使い分けは CLAUDE.md に書くだけでは切り替わらないため、`model:` を指定したサブエージェント定義で実現した

### 学んだこと
- CLAUDE.md はセッション開始時に毎回読み込まれる「常駐の指示書」であり、内容は短く・ルール中心に保つのが良い
- サブエージェント(`.claude/agents/*.md`)はフロントマターの `model:` でモデルを固定でき、役割ごとのモデル使い分けができる
